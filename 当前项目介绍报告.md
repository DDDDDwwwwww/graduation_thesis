## 2. 项目当前的整体定位

当前项目是一个围绕 **通用博弈游戏（GGP, General Game Playing）** 展开的实验型研究工程。它的目标不是只做某一个固定游戏的博弈程序，而是构建一套可以在多种 GDL/KIF 游戏规则上复用的搜索与价值评估框架，并在这个框架上实现你当前最终保留的方法线：

**Search-Distilled Residual Value** 及其配套实验体系。

从实际代码来看，项目现在已经形成了四层稳定结构：

1. **GGP 底座**
   - 负责读取 `games/*.kif` 中的游戏规则；
   - 把 GDL/KIF 规则翻译成 Prolog；
   - 用 SWI-Prolog 驱动状态机，统一提供合法动作、状态转移、终局判定和得分查询。

2. **MCTS 搜索层**
   - 提供纯搜索 baseline；
   - 提供带历史统计引导的启发式 MCTS baseline；
   - 提供把神经价值网络接入 MCTS 的统一外壳。

3. **价值模型训练层**
   - 支持早期的普通 value 训练；
   - 支持 teacher-only 训练；
   - 支持 residual_v1 训练；
   - 支持基于现有模型产物做推理加载。

4. **实验编排与结果产物层**
   - 支持数据生成、数据转换、模型训练、主 benchmark、baseline benchmark、ablation benchmark；
   - 把实验结果按照 `meta / raw / summary / metrics / artifacts / datasets / models` 的目录结构保存下来，形成比较完整的研究型工程输出。

因此，当前仓库已经不是一个“还在搭想法”的草稿仓库，而是一个**以 SDRPV 主线为中心、已经完成训练链路、评测链路和结果归档的实验工程**。

## 3. 当前项目实际在做什么

从现有实现来看，当前项目的核心任务可以概括为：

**在通用博弈游戏环境中，用搜索蒸馏得到的 teacher 信号和 baseline value 的残差学习，训练一个可接入 MCTS 的价值模型，并用多种 benchmark 验证其作为搜索增强组件的效果。**

把这句话拆开，当前仓库具体做了下面几件事。

### 3.1 用 GDL/KIF 规则驱动通用游戏状态机

项目不是手写某个游戏的规则逻辑，而是读取 `games/` 里的 KIF/GDL 描述，再通过：

- `src/gdl_parser.py`
- `src/ggp_statemachine.py`

把规则加载进 Prolog，形成统一的 `GameStateMachine`。

这意味着后续的搜索、对局、数据采样、模型评测，都建立在同一个通用状态机接口上，而不是每个游戏单独写一套环境。

### 3.2 在统一状态机上实现多种 MCTS agent

当前仓库实际保留并使用的 agent 有三类：

- `PureMCTAgent`
  - 纯 UCT + 随机 rollout；
  - 作为最基础 baseline。

- `HeuristicMCTSAgent`
  - 在统一 MCTS 核心上启用历史统计引导；
  - 作为比纯搜索更强的启发式 baseline。

- `NeuralValueMCTSAgent`
  - 把训练好的 value 模型接入 MCTS；
  - 支持 `value` 与 `selective` 两种 evaluator 模式；
  - 是当前 SDRPV 模型在线对局评测的统一执行外壳。

也就是说，当前项目不是单独训练一个 value 网络就结束，而是把它真正接回搜索系统中，形成“可实战评测”的 agent。

### 3.3 先完成旧 value 路线，再把旧资产转成 SDRPV 主线资产

从仓库结构和现有输出看，当前 SDRPV 不是凭空起步，而是建立在早期 value 学习实验之上的。

这一阶段的代表文件是：

- `experiments/generate_dataset.py`
- `experiments/train_value_model.py`
- `experiments/run_experiment_d_dataset_size.py`
- `outputs/experiments/D_dataset_size_sensitivity/...`

这一阶段做了两件很重要的事：

1. 先用自对弈和 baseline agent 生成普通 JSONL value 数据集；
2. 训练出一批早期的 baseline 模型产物，例如 `fact_mlp`、`token_mlp`、`token_transformer`。

这些旧资产没有被废弃，而是被当前 SDRPV 主线明确复用：

- 旧数据集被拿来转换成 SDRPV 数据集；
- 旧 token transformer 模型被拿来做 baseline benchmark 对照。

所以当前项目的真实演化路径是：

**先完成“通用 GGP + 普通 value 学习底座”，再在此基础上切换到 SDRPV 方法线。**

### 3.4 把旧数据转换为 SDRPV 数据格式

当前 SDRPV 数据入口是：

- `experiments/convert_dataset_to_sdrpv.py`

它把旧 JSONL 样本转换成当前主线使用的数据格式。转换后的每条样本包含：

- `s`
  - 状态信息，含 `state_facts`、`acting_role`、`ply_index`、`terminal`；
- `b`
  - cheap baseline value；
- `q_t`
  - teacher search value；
- `z`
  - 终局监督值；
- `phi`
  - phase-aware 特征；
- `metadata`
  - 数据来源与转换设定。

这一步对应的主要产物是：

- `outputs/datasets/sdrpv_dataset_v3_parallel.jsonl`
- `outputs/datasets/sdrpv_dataset_v3_parallel.jsonl.stats.json`
- `outputs/cache/sdrpv_baseline_cache_v3.jsonl`
- `outputs/cache/sdrpv_teacher_cache_v3.jsonl`

从工程角度看，这一步把“原始 value 样本”升级成了“可用于 teacher-only / residual 学习 / baseline 对照的 SDRPV 数据资产”。

### 3.5 在 SDRPV 数据上训练 teacher-only 与 residual_v1 模型

当前主线里最关键的两个训练入口是：

- `experiments/train_sdrpv_teacher_only.py`
- `experiments/train_sdrpv_residual_v1.py`

其中：

- teacher-only 训练负责直接拟合 `q_t` 或 `z`；
- residual_v1 训练负责学习 `target - b` 的残差，并在训练后输出离线比较指标。

对应的核心模型产物目录包括：

- `outputs/experiments/SDRPV_teacher_only_v3/`
- `outputs/experiments/SDRPV_residual_v1_residual_v1_gpu0_20260329_114156/`

这些目录中保存了：

- `model.pt`
- `encoder.json`
- `config.json`
- `metrics.json`
- `offline_metrics.json`（residual_v1 才有）

因此，当前项目的主方法实现不是停留在“设计了 residual 学习思想”，而是已经落成了完整的训练脚本与模型产物格式。

### 3.6 把训练好的模型重新接回 MCTS 做在线 benchmark

这一部分是当前项目已经真正完成的“方法落地”。

当前用于在线评测的主要脚本是：

- `experiments/run_sdrpv_residual_v1_mcts_grid.py`
- `experiments/run_sdrpv_residual_v1_main_benchmark.py`
- `experiments/run_sdrpv_residual_v1_baseline_benchmark.py`
- `experiments/run_sdrpv_ablation_benchmark.py`

它们不是只做离线 loss 对比，而是：

- 把训练好的模型加载成 `NeuralValueMCTSAgent`；
- 在真实对局里与 `pure_mct`、旧 baseline 模型、`heuristic_mcts` 等对手做对比；
- 分别记录 fixed-sims 和 fixed-time 两种预算下的对局结果、搜索统计和汇总文件。

这说明当前项目的完成度已经达到：

**模型训练 -> 模型封装 -> 搜索接入 -> 在线 benchmark -> 汇总产出**

这一整条闭环。

## 4. 这个项目是怎么一步步搭建出来的

结合仓库当前代码、输出文件命名和 `git log` 中保留下来的开发轨迹，可以把当前项目的形成过程概括为下面几个阶段。

### 4.1 第一阶段：先搭通用游戏运行底座

最早必须解决的问题，不是模型，而是“如何让不同游戏在同一套程序里跑起来”。

所以项目首先搭建了：

- `src/gdl_parser.py`
  - 把 GDL/KIF 文本解析成 S-Expression，并翻译成 Prolog 规则；
- `src/ggp_statemachine.py`
  - 在 SWI-Prolog 上实现通用游戏状态机。

有了这层之后，项目才具备了对不同游戏统一做：

- 初始状态读取；
- 合法动作生成；
- 联合动作执行；
- 下一个状态推演；
- 终局与目标值计算。

这一步是整个项目后面所有实验的共同起点。

### 4.2 第二阶段：搭出统一 MCTS 搜索框架

有了状态机以后，项目继续搭建搜索层：

- `src/mcts/tree_node.py`
- `src/mcts/selectors.py`
- `src/mcts/rollout.py`
- `src/mcts/evaluators.py`
- `src/agents/_mcts_core.py`

这一阶段的关键不是只写一个 agent，而是把 MCTS 的“选择、扩展、评估、回传”流程抽象出来，使得不同 agent 可以共用一套核心逻辑。

在这个基础上形成了：

- `PureMCTAgent`
- `HeuristicMCTSAgent`

这两类 baseline。

所以第二阶段的成果，是建立了“可跑对局、可作为 baseline 的 GGP 搜索系统”。

### 4.3 第三阶段：先完成普通 value 学习实验底座

在直接进入 SDRPV 之前，项目已经完成了一条普通 value 学习路线。对应代码主要是：

- `experiments/generate_dataset.py`
- `experiments/train_value_model.py`
- `experiments/run_experiment_d_dataset_size.py`
- `src/encoding/*`
- `src/nn/*`

这个阶段主要做了三件事：

1. 生成自对弈 JSONL 数据；
2. 训练多种表示/模型组合；
3. 保存可复用的 baseline 模型资产。

其中最有价值的遗产是：

- `outputs/experiments/D_dataset_size_sensitivity/datasets/dataset_*.jsonl`
- `outputs/experiments/D_dataset_size_sensitivity/models/*`

它们后来直接变成了 SDRPV 阶段的输入资产和对照资产。

所以从工程演化角度说，当前 SDRPV 项目不是推倒重来，而是**站在前一轮 value 学习工程之上升级出来的**。

### 4.4 第四阶段：切换到 SDRPV 数据转换与 teacher-only 路线

在普通 value 路线基础上，项目新增了：

- `experiments/convert_dataset_to_sdrpv.py`
- `experiments/train_sdrpv_teacher_only.py`

这一步的核心变化是：

- 不再只保留 `state -> value_target` 这种普通监督；
- 而是显式构造 `b`、`q_t`、`z`、`phi` 等 SDRPV 数据字段；
- 然后先跑一个 teacher-only 版本，验证 teacher 监督信号能不能学起来。

这一阶段的代表输出是：

- `outputs/datasets/sdrpv_dataset_v3_parallel.jsonl`
- `outputs/experiments/SDRPV_teacher_only_v3/`

也就是说，teacher-only 阶段在当前仓库中已经不是口头设想，而是一个独立完成、独立留档的阶段。

### 4.5 第五阶段：实现 residual_v1，形成当前主模型训练入口

在 teacher-only 成立之后，项目进入 residual 学习阶段，并新增：

- `experiments/train_sdrpv_residual_v1.py`

这个脚本是当前主模型训练链路的核心，因为它做了两件关键的工程化工作：

1. 用 `target - b` 构造残差监督；
2. 在训练后自动计算 `v_hat` 与 `b` 相比的离线比较指标，形成 `offline_metrics.json`。

这一步产出的目录：

- `outputs/experiments/SDRPV_residual_v1_residual_v1_gpu0_20260329_114156/`

标志着当前项目从“teacher-only 可训练”进一步推进到“residual 主线可训练、可离线比较”。

### 4.6 第六阶段：做 integration 小网格与主 benchmark，确定当前主用在线配置

在 residual 模型具备后，项目没有立即宣布最终方案，而是继续做接入方式比较。对应脚本包括：

- `experiments/run_sdrpv_residual_v1_mcts_grid.py`
- `experiments/run_sdrpv_residual_v1_main_benchmark.py`

这一阶段的目标是比较：

- `value_full`
- `selective`

这两类接入方式在多游戏 benchmark 下的表现。

这一阶段留下了：

- `outputs/experiments/SDRPV_residual_v1_mcts_grid_.../`
- `outputs/experiments/SDRPV_residual_v1_main_benchmark_.../`
- `sdrpv_main_benchmark_report.md`

从当前代码与产物看，**当前项目最终保留下来的主用在线配置是 residual_v1 + value_full**，而不是把 selective 接入继续当作最终主线推进。

这点很重要，因为它说明：

- 方案文档里提过的 selective integration 已经被实现并验证过；
- 但在当前仓库的“最终保留版本”里，它更多是一个已完成的对照分支，而不是当前主结果的中心。

### 4.7 第七阶段：做 baseline benchmark 和 ablation，补齐研究型实验结构

在主 benchmark 之后，项目继续完善了两类研究实验。

第一类是 baseline 对比：

- `experiments/run_sdrpv_residual_v1_baseline_benchmark.py`

它把当前主模型与：

- `pure_mct`
- 旧 `token_transformer` baseline
- `heuristic_mcts`

做统一 benchmark。

第二类是消融实验：

- `experiments/run_sdrpv_ablation_train.py`
- `experiments/run_sdrpv_ablation_benchmark.py`

它把当前方法拆成：

- full residual + teacher
- no residual + teacher
- no teacher + residual

三个版本，在相同 MCTS 设置下进行比较。

这两部分说明当前项目已经从“方法实现”进一步补到了“论文型实验结构”。

### 4.8 第八阶段：围绕 SDRPV 主线清理仓库

从近期提交信息可以看出，项目最后还做了一轮围绕 SDRPV 主线的清理和依赖梳理。

这一阶段的目的不是增加新方法，而是让仓库结构更清晰：

- 以 SDRPV 为主线保留必要文件；
- 删除或弱化不再作为主线的早期遗留内容；
- 保留旧资产中仍被当前主线复用的部分。

所以当前仓库的状态，已经是你所说的“终极版本”：不是还在试验方向，而是已经完成主线实现后再做过收束整理的版本。

## 5. 当前项目的主方法到底是什么

基于当前实际实现，而不是方案原始设想，当前项目的主方法可以表述为：

**基于 GGP 通用状态机和统一 MCTS 框架，使用 SDRPV 数据集训练 residual value 模型，再以 `value_full` 方式接入 MCTS，完成跨游戏 benchmark、baseline benchmark 和 ablation benchmark。**

更具体地说，当前主方法由下面几部分组成。

### 5.1 输入表示

当前主用表示来自：

- `src/encoding/board_token_encoder.py`
- `src/encoding/board_token_mlp_encoder.py`

主线更偏向：

- `board_token`
- `transformer`

其中 `BoardTokenEncoder` 会把 `cell(x,y,content)` 事实编码为 token 序列，并附带一些轻量全局特征，例如：

- 当前玩家信息；
- 步数进度；
- 是否终局。

### 5.2 模型结构

当前神经网络定义在：

- `src/nn/value_net.py`

支持两类主结构：

- `MLPValueNet`
- `TransformerValueNet`

而当前 SDRPV 主线实际使用的是：

- `board_token + transformer`

这点可以从现有 `config.json` 和实验脚本默认参数中直接看出来。

### 5.3 训练目标

当前训练入口有两类：

- teacher-only
  - 直接学习 `q_t` 或 `z`；
- residual_v1
  - 学习 `target - b` 的残差。

在当前终极版本里，真正成为主方法的是：

- `train_sdrpv_residual_v1.py`

也就是 residual 路线，而 teacher-only 更像是主线推进中的前一阶段和对照阶段。

### 5.4 在线接入方式

虽然代码里保留了 `value` 和 `selective` 两种 evaluator 模式，但从当前保留下来的主 benchmark 结论和 baseline benchmark 入口设计看，当前主用在线配置是：

- `NeuralValueMCTSAgent`
- `evaluator_mode = value`

也就是当前项目中的 `value_full` 路线。

### 5.5 当前没有成为主线的内容

从实际代码状态看，下面这些内容虽然在方案中出现过，但**没有成为当前最终主线的中心实现**：

- `phase-aware` 没有成为当前主 benchmark 的主方法
  - `run_sdrpv_residual_v1_main_benchmark.py` 的文档字符串已明确写明是 “without phase-aware”；
  - 数据转换脚本虽然会生成 `phi` 字段，但当前 teacher-only / residual_v1 训练入口并没有把 `phi` 单独作为主输入分支接入模型。

- `selective integration` 已实现，但不是当前最终保留的主配置
  - 相关 evaluator 与 benchmark 都已完成；
  - 但当前主方法定位已经回到 `value_full`。

因此，当前项目的真实最终形态不是“完整实现了方案中的全部增强项”，而是：

**实现了方案里最核心、并且最终被你保留下来的那一部分。**

## 6. 目录与文件作用说明

下面按目录说明当前项目中各类文件分别是什么、从哪里来、在主线里起什么作用。

### 6.1 根目录文件

#### `search_distilled_residual_value_plan.md`

这是方法设计方案文档，用来定义 SDRPV 的原始设想和阶段推进路线。

它的作用是“设计蓝图”，不是“当前实现事实”。当前仓库中的很多实现确实从这里出发，但并不是所有设想都落成了最终主线。

#### `sdrpv_main_benchmark_report.md`

这是主 benchmark 阶段的总结报告，对应 residual 模型接回 MCTS 后的主实验。

它在当前仓库里代表“主方法线上比较阶段”的文字归档。

#### `offline sanity check.md`

这是围绕 `offline_metrics.json` 进一步整理出来的离线 sanity check 文档。

它的作用不是替代训练产物中的原始指标文件，而是把 `b / teacher-only / residual` 三种离线 value 结果统一整理成可直接引用的表格，并补上一段面向论文写作的结果解释。

### 6.2 `games/`

这个目录存放 GDL/KIF 游戏规则文件，例如：

- `games/connectFour.kif`
- `games/breakthrough.kif`
- `games/hex.kif`
- `games/ticTacToe.kif`

这些文件是整个项目最底层的数据源之一，因为状态机就是从这里读规则。

### 6.3 `prolog/`

这个目录存放由 GDL/KIF 翻译得到的 Prolog 文件，例如：

- `prolog/connectFour.pl`
- `prolog/breakthrough.pl`

它们来自 `src/gdl_parser.py` 的翻译结果，是 GGP 规则落到 SWI-Prolog 执行层后的中间形式。

### 6.4 `src/`

这是当前项目的核心源码目录。

#### `src/gdl_parser.py`

负责：

- 解析 GDL/KIF；
- 生成 Prolog 规则。

它解决的是“规则怎么读进来”的问题。

#### `src/ggp_statemachine.py`

负责：

- 加载规则；
- 构造 `GameStateMachine`；
- 提供 `get_roles / get_initial_state / get_legal_moves / get_next_state / is_terminal / get_goal` 等统一接口；

- 记录一定的性能统计。

它是所有 agent、数据生成脚本、benchmark 脚本的共同依赖。

#### `src/mcts/`

这一层是搜索基础设施。

- `tree_node.py`
  - 定义搜索树节点结构；
- `selectors.py`
  - 实现 UCT 选择和 Gibbs 采样；
- `rollout.py`
  - 实现 rollout 逻辑；
- `evaluators.py`
  - 定义随机 rollout、启发式 rollout、value 网络评估、selective 评估等叶节点评估器。

#### `src/agents/`

这一层是可执行的搜索/对局 agent 封装。

- `base_agent.py`
  - agent 基类；
- `_mcts_core.py`
  - 统一 MCTS 主循环；
- `pure_mct_agent.py`
  - 纯 MCTS baseline；
- `heuristic_mcts_agent.py`
  - 启发式 MCTS baseline；
- `neural_value_mcts_agent.py`
  - 神经价值模型接入 MCTS 的统一 agent。

#### `src/encoding/`

这一层负责把游戏状态转成模型输入。

- `board_token_encoder.py`
  - 把棋盘事实编码成 token 序列；
- `board_token_mlp_encoder.py`
  - 把 board token 结构进一步整理成 MLP 可用输入；
- `vocab.py`
  - fact vocabulary；
- `__init__.py`
  - 模块导出。

#### `src/nn/`

这一层负责模型训练和推理。

- `value_net.py`
  - 定义 MLP 与 Transformer value 网络；
- `dataset.py`
  - Dataset 和 batch 组织；
- `trainer.py`
  - 统一训练循环；
- `inference.py`
  - 模型/编码器产物加载与推理。

### 6.5 `experiments/`

这是当前项目最重要的实验入口目录。

#### 前期普通 value 路线

- `generate_dataset.py`

  - 训练普通 value 模型；

#### SDRPV 评测主线

#### 公共实验基础设施

- `experiment_utils.py`
  - 提供输出目录初始化、子命令运行、match grid 执行、汇总文件写出等公共能力。

### 6.6 `scripts/`

这是 shell 层的一键编排入口。

## 7. 当前项目中的数据、模型和结果文件是怎么来的

这一部分专门回答“各种脚本输出是用来做什么的、怎么来的”。

### 7.1 早期普通 value 数据集与模型怎么来的

来源链路是：

1. `experiments/generate_dataset.py`
   - 用 `pure_mct` / `heuristic_mcts` 做自对弈；
   - 采样中间状态；
   - 写成普通 JSONL value 样本。

2. `experiments/train_value_model.py`
   - 读取这些普通 JSONL 数据集；
   - 训练 `fact_mlp`、`token_mlp`、`token_transformer` 等模型。

3. `experiments/run_experiment_d_dataset_size.py`
   - 自动化完成不同数据规模下的数据生成、模型训练和对局评测；
   - 将结果保存到 `outputs/experiments/D_dataset_size_sensitivity/`。

所以：

- `outputs/experiments/D_dataset_size_sensitivity/datasets/dataset_*.jsonl`
  - 是早期自对弈 value 数据集；
- `outputs/experiments/D_dataset_size_sensitivity/models/*`
  - 是早期 value 模型产物；
- `artifact_manifest.json`
  - 记录不同数据规模、不同模型配置对应的产物路径。

### 7.2 当前 SDRPV 数据集怎么来的

来源链路是：

1. 读取旧数据集，例如：
   - `outputs/experiments/D_dataset_size_sensitivity/datasets/dataset_200.jsonl`
2. 运行：
   - `experiments/convert_dataset_to_sdrpv.py`
3. 生成：
   - `outputs/datasets/sdrpv_dataset_v3_parallel.jsonl`
   - `outputs/datasets/sdrpv_dataset_v3_parallel.jsonl.stats.json`
   - `outputs/cache/sdrpv_baseline_cache_v3.jsonl`
   - `outputs/cache/sdrpv_teacher_cache_v3.jsonl`

其中：

- `baseline_cache`
  - 记录某些状态的 baseline 值，避免重复计算；
- `teacher_cache`
  - 记录某些状态的 teacher 搜索值，避免重复计算；
- `stats.json`
  - 记录本轮转换的成功率和 `b/q_t/z/|q_t-b|` 的统计概况。

### 7.3 当前主模型怎么来的

来源链路是：

1. 读取 `outputs/datasets/sdrpv_dataset_v3_parallel.jsonl`
2. 运行：
   - `experiments/train_sdrpv_teacher_only.py`
   - 或 `experiments/train_sdrpv_residual_v1.py`
3. 生成模型目录，例如：
   - `outputs/experiments/SDRPV_teacher_only_v3/`
   - `outputs/experiments/SDRPV_residual_v1_residual_v1_gpu0_20260329_114156/`

每个模型目录通常包含：

- `model.pt`
  - 模型权重；
- `encoder.json`
  - 编码器配置；
- `vocab.json`
  - 仅 fact-vector 路线需要；
- `metrics.json`
  - 训练过程中常规验证指标；
- `config.json`
  - 本轮训练参数与样本规模说明；
- `offline_metrics.json`
  - residual_v1 额外输出的离线比较指标。

### 7.4 主 benchmark、baseline benchmark 和 ablation 输出怎么来的

来源链路是：

1. 加载模型目录中的 `model.pt + encoder.json`；
2. 用 `NeuralValueMCTSAgent.from_artifacts(...)` 封装成可对局 agent；
3. 通过 `run_match_grid(...)` 在多个游戏、多个 seed、fixed-sims 与 fixed-time 两种预算下运行；
4. 将结果按结构化目录写入 `outputs/experiments/...`。

因此，类似下面这些目录：

- `outputs/experiments/SDRPV_residual_v1_main_benchmark_.../`
- `outputs/experiments/SDRPV_residual_v1_baseline_benchmark/`
- `outputs/experiments/SDRPV_ablation_train_.../`
- `outputs/experiments/SDRPV_ablation_benchmark_.../`

都属于“训练后的在线评测或批量比较”产物。

## 8. 各类输出文件分别记录什么

当前项目的结果目录结构非常统一，这一点是工程上比较成熟的部分。

### 8.1 `config.json`

记录一次训练或一次实验的配置参数，例如：

- 数据集来源；
- 编码器与模型类型；
- epochs、batch size、learning rate；
- seed；
- 训练样本数；
- 某些关键测试指标。

它的作用是：让模型或实验结果能追溯回“当时是怎么跑出来的”。

### 8.2 `metrics.json`

记录训练过程中的常规指标，例如：

- test loss；
- sign accuracy；
- best validation loss 等。

它主要属于模型训练阶段的标准输出。

### 8.3 `offline_metrics.json`

这是 residual_v1 训练阶段最有特征的一类输出。

它记录的是：

- `v_hat` 与目标值的相关性；
- `b` 与目标值的相关性；
- `MAE(v_hat, target)` 与 `MAE(b, target)` 的比较。

它的作用不是最终对局结论，而是说明 residual 学习是否在离线层面真正超过 baseline evaluator。

在当前仓库里，这类原始离线指标还被进一步整理成了根目录下的 `offline sanity check.md`。后者把 `offline_metrics.json` 中 residual 对 baseline 的比较，与保存下来的 teacher-only checkpoint 重算结果放到同一张表里，用统一口径补出了 `MAE / MSE / Corr` 三项指标，便于在文字报告中直接引用。

### 8.4 `run_manifest.json`

这是实验运行层的元信息文件，通常放在 `meta/` 下。

它会记录：

- 实验名；
- 输出根目录；
- 本轮运行的命令参数。

它的作用类似“实验执行登记表”。

### 8.5 `matches.json`

这是 `raw/` 目录下的原始对局级结果，保留每场 match 的详细信息。

它是后续聚合统计的底层数据。

### 8.6 `by_game.json / by_game.csv`

这是按游戏汇总的结果。

它适合回答：

- 在某一个具体游戏上，当前 agent 表现如何；
- 决策耗时和评估调用统计怎样。

### 8.7 `per_seed_summary.json / csv`

这是按随机种子汇总的结果。

它适合回答：

- 不同 seed 下结果是否稳定；
- 某种配置是否只是在个别 seed 上有效。

### 8.8 `aggregate_summary.json / csv`

这是最终聚合结果，一般用于跨游戏、跨 seed 总结。

它更接近实验报告里引用的“总表”。

### 8.9 `delta_vs_full.json / csv`

这是消融 benchmark 的对比汇总文件。

它记录某个 ablation 配置相对于 full 版本的差值。

### 8.10 `artifact_manifest.json`

这是模型产物清单文件。

它最典型地出现在 `D_dataset_size_sensitivity` 中，用来记录不同实验配置生成了哪些模型文件。

## 9. 当前项目的主流程是怎样串起来的

如果把当前仓库真正能跑通的主流程按顺序写出来，大致是下面这样：

1. 在 `games/*.kif` 中定义游戏规则。
2. 用 `src/gdl_parser.py` 和 `src/ggp_statemachine.py` 把规则转成可执行的通用状态机。
3. 用 `PureMCTAgent`、`HeuristicMCTSAgent` 等 baseline agent 运行自对弈，生成早期普通 value 数据。
4. 用 `run_experiment_d_dataset_size.py` 训练早期 value 模型，并形成可复用旧资产。
5. 用 `convert_dataset_to_sdrpv.py` 把旧 JSONL 数据转换成 SDRPV 数据集。
6. 用 `train_sdrpv_teacher_only.py` 验证 teacher-only 路线。
7. 用 `train_sdrpv_residual_v1.py` 训练当前主模型 residual_v1，并导出离线指标。
8. 用 `run_sdrpv_residual_v1_mcts_grid.py` 做小范围 integration 配置筛选。
9. 用 `run_sdrpv_residual_v1_main_benchmark.py` 做当前主方法的线上主 benchmark。
10. 用 `run_sdrpv_residual_v1_baseline_benchmark.py` 做当前主模型与 baseline 的正式对比。
11. 用 `run_sdrpv_ablation_train.py` 和 `run_sdrpv_ablation_benchmark.py` 做消融实验。
12. 把所有输出归档到 `outputs/` 下，并用根目录报告文件做阶段性文字总结。

这个流程说明，当前项目已经形成了**从规则、搜索、数据、训练、评测到归档**的完整研究闭环。

## 10. 当前项目的真正研究目的

基于现有仓库，当前项目的研究目的可以准确地表述为：

### 10.1 方法层目的

探索一种在 GGP 场景下可行的神经价值增强路径，不再只依赖普通终局监督，而是利用：

- teacher search 信号；
- baseline value；
- residual 学习；
- MCTS 接入比较；

来构造比单纯 baseline 更强的搜索增强方法。

### 10.2 工程层目的

把这种方法做成一个真正能复现实验的工程，而不是停留在单个 notebook 或单轮脚本。

这也是为什么当前项目里会同时保留：

- 统一状态机；
- 统一 agent 框架；
- 标准化模型产物；
- 标准化 benchmark 输出结构；
- 标准化实验编排脚本。

### 10.3 论文层目的

从仓库结构已经可以明显看出，这个项目不是单纯为了“把程序跑起来”，而是为了支撑毕业论文式的研究叙事：

- 有方法设计文档；
- 有阶段性实验报告；
- 有主 benchmark；
- 有 baseline benchmark；
- 有 ablation；
- 有产物归档与结果汇总结构。

因此，当前项目本质上是一个**以 GGP 搜索增强为主题、以 SDRPV 为主线、面向论文产出的实验研究工程**。

## 11. 对当前终极版本的准确概括

如果要用几句话准确概括你现在这个项目，可以写成下面这样：

这个项目首先搭建了一套基于 GDL/KIF 与 SWI-Prolog 的通用博弈游戏执行底座，并在其上实现了统一的 MCTS 搜索框架、baseline agent 和神经价值接入 agent。随后，项目先完成了普通 value 学习的数据与模型基础设施，再把这些旧资产转换并升级为 SDRPV 主线所需的数据、缓存与模型体系，依次完成了 teacher-only、residual_v1、integration benchmark、baseline benchmark 与 ablation benchmark。当前仓库保留下来的最终主线，不是方案文档中的全部设想，而是其中真正被实现并最终保留下来的部分：以 residual_v1 模型为核心、以 `value_full` 作为主要在线接入方式的 SDRPV 实验工程。

换句话说，当前仓库已经完成的不是某个局部技巧，而是一条完整的方法实现链：

**GGP 规则执行 -> MCTS 搜索 -> 数据生成与转换 -> SDRPV 训练 -> 神经搜索接入 -> 多层 benchmark 与消融 -> 结果归档。**

这就是当前项目“终极版本”的真实形态。
