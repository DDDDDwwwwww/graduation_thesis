## 方案 C：Token Transformer + 全局特征融合网络（Token-Global Fusion Value Net）

### 1. 目标

在现有 `neural_mcts:token_transformer` 基线之上，设计一个**Token 表示 + 全局特征融合**的价值网络，以解决当前实验中暴露出的两个关键问题：

- `token_transformer` 在复杂游戏上有潜力，但在简单游戏或小数据条件下表现不稳定；在你的结果里，`neural_mcts:token_transformer` 对 `pure_mct` 在 `breakthrough` 上胜率为 0.7，但在 `ticTacToe` 和 `connectFour` 上分别只有 0.0 和 0.3。:contentReference[oaicite:0]{index=0}
- `token_transformer` 对数据量比较敏感，数据从 200 提高到 2000 后，`value_greedy:token_transformer` 对 random 的胜率从 0.30 提升到 0.80，`neural_mcts:token_transformer` 也从 0.70 提升到 1.00。:contentReference[oaicite:1]{index=1}

因此，本方案的核心思想是：

> 保留 `BoardTokenEncoder + TransformerValueNet` 对棋盘局面的高表达能力，同时引入一组低维、廉价、跨游戏尽量通用的全局统计特征，提升小样本条件下的学习稳定性和样本效率。

---

### 2. 方法总览

本方案的最终模型记为：

> **Token-Global Fusion Value Net**

整体结构分为三部分：

1. **Token 分支**：使用 `BoardTokenEncoder` 和 `TransformerValueNet` 对棋盘局面进行细粒度表示学习；
2. **Global 分支**：从当前局面中提取一组全局统计特征，并用小型 MLP 编码；
3. **融合头（Fusion Head）**：将 token 表示与 global 表示拼接后，经若干线性层输出最终价值。

该网络可用于两种 agent 形式：

- `value_greedy:token_transformer_global`
- `neural_mcts:token_transformer_global`

其中，后者是最主要的论文方法。

---

### 3. 方法动机

当前实验已经给出三个对本方案有直接支持的现象：

#### 3.1 token_transformer 有表达潜力，但并不总是稳定
在多游戏结果中，`neural_mcts:token_transformer` 对 `pure_mct` 在 `breakthrough` 上胜率达到 0.7，但在 `ticTacToe` 和 `connectFour` 上表现明显较差，说明纯 token-transformer 路线在复杂局面上有潜力，但并非普适优势。:contentReference[oaicite:2]{index=2}

#### 3.2 token_transformer 更依赖数据规模
在数据规模实验 D 中，`value_greedy:token_transformer` 随着数据从 200 增加到 2000，胜率由 0.30 提高到 0.80，提升非常明显；这说明其表示能力较强，但需要更多数据来稳定学习。:contentReference[oaicite:3]{index=3}

#### 3.3 fact_mlp 更稳健，说明低维全局结构信息是有价值的
在编码器/模型消融中，`value_greedy` 对 random 的最好配置是 `fact_mlp`，而不是 `token_transformer`，说明低维统计/结构特征在小数据或简单评估场景下更稳健。:contentReference[oaicite:4]{index=4}

因此，本方案的假设是：

> `token_transformer` 的优势在于细粒度局面表示，而 `fact_mlp` 一类方法的优势在于低维全局先验。若将两者结合，有望同时获得“表达能力”和“样本效率”。

---

### 4. 模型结构

## 4.1 输入组成

模型输入由两部分组成：

### A. Token 输入
由 `BoardTokenEncoder` 负责生成，例如：

- `tile_content_ids`
- `tile_positions`
- `mask`

其作用是让模型学习棋盘局面的细粒度空间结构与内容模式。

### B. Global Features 输入
由新的全局特征提取模块负责生成，例如：

- 当前玩家
- 当前合法动作数
- 当前回合数或步数
- 棋盘占用率
- 双方棋子数量
- 双方棋子数量差
- 各类关键内容 token 的计数
- 终局接近程度的粗指标

这些特征应尽量满足：

- 易于从状态中提取
- 低维
- 不强依赖某一个具体游戏的特定规则
- 对不同棋盘游戏尽可能通用

---

## 4.2 Token 分支

Token 分支保留你现有的主干思路：

1. 使用 `BoardTokenEncoder` 将局面编码为 token 序列；
2. 对 token 做嵌入与位置编码；
3. 输入 `TransformerValueNet`；
4. 使用 pooling（如 mean pooling / cls-like pooling）得到 token-level 全局表示：
\[
h_{\text{token}} \in \mathbb{R}^{d_t}
\]

设计要求：

- 尽量复用现有 `token_transformer` 代码；
- 初版不要大改 transformer 结构，保证与 baseline 只在“是否加入 global 分支”上形成清晰对照。

---

## 4.3 Global 分支

Global 分支负责处理局面的低维统计特征。

输入：
\[
g(s) \in \mathbb{R}^{d_g}
\]

其中 `d_g` 为全局特征维度。

通过一个小型 MLP 编码：
\[
h_{\text{global}} = \text{MLP}(g(s)) \in \mathbb{R}^{d_h}
\]

推荐结构：

- 1～2 层全连接
- ReLU / GELU
- 维度不宜过大，例如 `32` 或 `64`

设计原则：

- Global 分支必须足够轻量
- 它的作用不是替代 transformer，而是提供廉价、稳定的先验补充

---

## 4.4 融合头（Fusion Head）

将两路特征拼接：

\[
h = [h_{\text{token}} ; h_{\text{global}}]
\]

再送入一个小型输出头得到最终价值：

\[
v(s) = \sigma(\text{MLP}_{\text{head}}(h))
\]

其中：

- `v(s) ∈ [0,1]`
- 表示当前局面对当前玩家的价值估计

推荐实现：

- 拼接后接 1～2 层 MLP
- 最后一层输出 1 维
- 用 sigmoid 映射到 `[0,1]`

---

### 5. 全局特征设计

本方案成功与否，很大程度取决于全局特征是否合理。为控制复杂度，建议从**少量稳定特征**开始，而不是一开始堆很多特征。

---

## 5.1 第一类：玩家与行动空间特征

这些特征通常跨游戏较通用。

建议包括：

- `current_player_id`
- `num_legal_actions`
- `normalized_num_legal_actions`
- 上一轮与当前轮合法动作数差值（若实现方便）
- 是否存在立即终结动作的粗指标（可选）

作用：

- 反映当前行动复杂度
- 帮助模型快速判断局面是否“有很多可选路线”或“接近强制走法”

---

## 5.2 第二类：局面规模与进度特征

建议包括：

- 当前回合数 / move count
- 归一化步数
- 棋盘占用格数
- 棋盘占用率
- 空格比例

作用：

- 让模型知道当前处于开局、中局还是残局附近
- 补足纯 token 表示可能需要较多样本才能隐式学出的“局面阶段感”

---

## 5.3 第三类：棋子/内容统计特征

针对棋盘类游戏，通常可以从 token 内容中统计一些分布特征。

建议包括：

- 当前玩家棋子数
- 对手棋子数
- 双方棋子差
- 各主要内容 token 的计数
- 各主要内容 token 的归一化频率

作用：

- 让模型更快获得“物质优势/劣势”的粗线索
- 在小数据时提升稳定性

---

## 5.4 第四类：游戏结构粗特征（可选）

若实现不困难，可额外加入：

- 棋盘宽度
- 棋盘高度
- 棋盘总格数
- 是否是双人零和（若系统已有）
- 是否有明显重力机制（如 connectFour，若已有通用检测可加，否则先不做）

这些特征可帮助模型适应不同棋盘拓扑，但初版不是必须。

---

### 6. 推荐特征集合（初版）

为避免特征工程过重，建议初版先使用一个**小而稳**的全局特征集合：

1. 当前玩家 one-hot
2. 当前合法动作数
3. 当前回合数 / move count
4. 棋盘占用率
5. 当前玩家棋子数
6. 对手棋子数
7. 双方棋子数量差
8. 非空格数量
9. 棋盘总格数
10. 空格比例

这个版本已经足以支撑第一轮实验。

---

### 7. 编码器与接口设计

建议在现有 `BoardTokenEncoder` 基础上扩展，使其能够同时输出 token 输入和 global features。

推荐接口如下：

```python
class BoardTokenEncoder:
    def encode(self, state, game, role=None) -> dict:
        """
        返回:
        {
            "tile_content_ids": np.ndarray [T],
            "tile_positions": np.ndarray [T] or [T, 2],
            "global_features": np.ndarray [G],
            "mask": np.ndarray [T]
        }
        """
```

其中：

* `tile_content_ids`、`tile_positions`、`mask` 继续服务于 token 分支
* `global_features` 供 global 分支使用

这样做的优点：

* 与现有架构兼容
* 结构清晰
* 方便以后继续加特征

---

### 8. 训练方案

## 8.1 总体原则

本方案不引入新的训练目标，仍沿用你当前价值网络训练框架：

* 输入状态 `s`
* 输出 value 预测
* 目标为局面最终结果或你当前使用的价值标签

这样可以保证：

* 与 baseline 对比公平
* 变量集中在“模型结构变化”上
* 更容易复用现有训练代码和数据

---

## 8.2 训练数据

推荐直接复用当前已有训练数据生成方式，包括：

* 自博弈生成的状态-结果样本
* 现有离线数据集

重点是保证：

* baseline `token_transformer`
* 新模型 `token_transformer_global`

使用相同数据集、相同划分方式、相同训练轮数区间

---

## 8.3 训练目标

与原始 value net 一致，例如：

* 二分类风格结果预测
* 或标量价值回归

若你当前系统输出的是 `[0,1]` 胜率风格 value，则继续保持一致即可。

---

## 8.4 控制变量要求

为了让实验结论更清楚，建议控制以下变量不变：

* 训练数据规模
* batch size
* optimizer
* learning rate（初版）
* 训练 epoch 数
* 自博弈数据来源
* MCTS 主体逻辑

只改变：

* 是否加入 global 分支
* 加入哪些 global features

### 10. 实验目标

本方案的实验需要回答以下问题：

1. 在相同训练数据下，加入 global features 是否能提升 `token_transformer` 的稳定性？
2. 这种改进是否主要体现在**小数据 regime**？
3. 融合模型是否优于 token-only 和 global-only？
4. 在真实对战中，融合模型是否能提升 `neural_mcts` 的整体胜率？

---

## 11. 实验一：数据规模敏感性实验（核心实验）

### 11.1 目的

这是本方案最重要的实验，因为你的方法动机本质上是：

> 通过加入低维全局先验，提高 token-transformer 在小数据条件下的样本效率。

当前实验 D 已经证明 `token_transformer` 对数据规模非常敏感。
因此，这个实验必须作为主实验之一。

---

### 11.2 对比方法

建议至少包括：

* `value_greedy:fact_mlp`
* `value_greedy:token_transformer`
* `value_greedy:token_transformer_global`
* `neural_mcts:fact_mlp`
* `neural_mcts:token_transformer`
* `neural_mcts:token_transformer_global`

可选：

* `neural_mcts:token_mlp`

---

### 11.3 推荐设置

先完全复用现有 D 实验框架：

* 游戏：`connectFour`
* 数据规模：`[200, 500, 1000, 2000]`
* 每组对局数：`10` 局起步
* 对战 agent：`random`

你当前 D 就是这个设置。

---

### 11.4 重点分析

重点看以下问题：

* 在 `200` 和 `500` 这类低数据条件下，`token_transformer_global` 是否优于 `token_transformer`
* 它是否缩小了与 `fact_mlp` 的稳定性差距
* 当数据增大到 `1000`、`2000` 时，是否仍能保持不劣于 baseline

---

### 11.5 推荐扩展

如果资源允许，建议在 `breakthrough` 上再补一组小规模数据实验。

原因：

* 你当前 D 只在 `connectFour` 上做，结论偏单一
* 而 H 显示 `breakthrough` 更能体现神经方法潜力。

可采用更轻量设置：

* 游戏：`breakthrough`
* 数据规模：`[200, 500, 1000]`
* 只比较：

  * `neural_mcts:token_transformer`
  * `neural_mcts:token_transformer_global`

---

## 12. 实验二：主基准对战实验

### 12.1 目的

---

### 12.2 对比方法

至少包括：

* `pure_mct`
* baseline：`neural_mcts:token_transformer`
* 新方法：`neural_mcts:token_transformer_global`

### 12.3 推荐设置

复用你当前 A/H 主基准设置：

* 游戏：

  * `ticTacToe`

### 12.4 关键比较

重点比较：

* `neural_mcts:token_transformer_global vs pure_mct`
* `neural_mcts:token_transformer_global vs neural_mcts:token_transformer`

目标是验证：

* 在 `breakthrough` 上保持或扩大 baseline 优势
* 在 `ticTacToe` / `connectFour` 上至少比 baseline 更稳一些

---

## 13. 实验三：融合方式消融实验

### 13.1 目的

验证性能提升是否真的来自“token + global 融合”，而不是其中某一支路单独更强。

---

### 13.2 对比版本

建议至少比较以下三种：

1. **Token-only**

   * `BoardTokenEncoder + TransformerValueNet`
   * 即 baseline `token_transformer`

2. **Global-only**

   * 仅使用 `global_features + MLP`
   * 这是新的 global-only 版本

3. **Token + Global Fusion**

   * 本方案主模型

---

### 13.3 推荐设置

优先在：

* `connectFour`
* 数据规模：`200 / 500 / 1000`
* `value_greedy` 与 `neural_mcts` 各跑一组

如果预算不足，可优先只做 `neural_mcts`。

---

### 13.4 结论目标

应当展示：

* Global-only 往往更稳，但表达能力有限
* Token-only 潜力高，但小数据不稳
* Fusion 在两者之间达到更好的平衡

---

## 14. 实验四：特征组消融实验

### 14.1 目的

分析到底是哪一类全局特征带来了帮助，避免论文看起来像“盲目堆特征”。

---

### 14.2 推荐分组

可将全局特征分为三组：

* **Group A：玩家与动作空间特征**
* **Group B：局面进度与占用率特征**
* **Group C：棋子/内容统计特征**

比较以下版本：

1. Token + A
2. Token + B
3. Token + C
4. Token + A+B+C（完整版本）

---

### 14.3 推荐设置

* 游戏：`connectFour`
* 数据规模：`500`
* agent：优先 `value_greedy` 或 `neural_mcts` 其一

选择中等数据量，是因为太小会噪声太大，太大又容易掩盖差异。

---

## 15. 实验五：时间预算敏感性实验（可选）

### 15.2 推荐设置

* `playclock ∈ {0.2, 0.7, 1.5}`
* `iterations = 180`
* 对手：`random`
* 比较：

  * `neural_mcts:token_transformer`
  * `neural_mcts:token_transformer_global`

你当前 B 就是这个框架。

### 15.3 说明

这个实验对方案 C 不是必须主实验，但它可以作为补充证据，证明融合模型在有限预算下也更稳。

* 原始 `token_transformer` 结果不能直接算作新模型结果
* 但它们可以作为 baseline 对照和方法动机
* 新模型必须重新训练并重新评测

## 19.3 新增工作量

本方案新增工作主要包括：

1. 扩展 `BoardTokenEncoder` 输出 `global_features`
2. 实现 `GlobalFeatureMLP`
3. 实现 `Token-Global Fusion Value Net`
4. 重新训练融合模型
5. 增加融合消融与特征组消融实验

总体上，这个方案的新增工作量**大于方案 B**，但“模型创新感”更强。

### 20. 推荐实施顺序

## 第一阶段：最小可运行版本

先做最小闭环：

* 只选 5～10 个全局特征
* 只实现最简单的拼接融合
* 先只做 `value_greedy:token_transformer_global`
* 在 `connectFour` 上跑 D 实验中的 `200 / 500 / 1000`

目标：

* 快速判断融合方向是否有效

若第一阶段有效，再扩展到：

* `neural_mcts:token_transformer_global`

* 跑 A/H 主基准对战

## 第三阶段：论文完善版本

最后再做：

* 融合消融
* 特征组消融

### 21. 推荐默认配置

若需要先快速落地一个默认版本，建议如下：

* Token 分支：当前 `BoardTokenEncoder + TransformerValueNet`
* Global 分支：2 层 MLP，hidden dimension = `32`
* 融合方式：拼接后 1 层 MLP 输出
* 全局特征：10 维左右
* 首个测试游戏：`connectFour`
* 首个主实验：数据规模 `[200, 500, 1000]`

这个版本实现简单，足以支持第一轮判断。

### 22. 论文中可使用的创新表述

可将本方案的创新概括为：

> 针对 General Game Playing 中 token-transformer 价值网络在小数据条件下学习不稳定的问题，提出一种融合棋盘 token 表示与全局统计特征的价值网络。该方法在保留 token-transformer 细粒度局面建模能力的同时，引入低维全局先验，从而提升样本效率与跨游戏稳定性。

如果要写得更工程一点，也可以表述为：

> 提出一种 Token-Global Fusion Value Net，用于改进 `neural_mcts:token_transformer` 在多游戏、小样本设置下的稳定性。

---

### 23. 方案总结

本方案的核心不是简单“加几个特征”，而是围绕当前实验里最明确的问题做结构性改进：

* `token_transformer` 有潜力，但不够稳；
* `fact_mlp` 稳，但表达上限不一定高；
* 因此，用全局低维特征去补足 token-transformer 的小样本短板。

从论文角度看，本方案的优点是：

* 模型创新比单纯调参更明确
* 与现有 baseline 关系清晰
* 能直接对应你当前实验里的数据规模敏感性现象
* 可复用现有 D、A/H、E 实验框架

因此，若你希望论文的创新点更偏“新网络结构”，而不是“搜索调用策略”，本方案是一个合理且可写的方向。
